## 2. ID生成规则

### 2.1 任务编号 (task\_no) 生成规则

**格式：** `TASK{yyyyMMddHHmmss}{毫秒内序号}{随机字符}`

**示例：** `TASK2026050212345612345A3F`

**组成部分：**

| 部分             | 长度 | 说明              |
| -------------- | -- | --------------- |
| TASK           | 4  | 固定前缀            |
| yyyyMMddHHmmss | 14 | 年月日时分秒          |
| 毫秒内序号          | 6  | 同一毫秒内的自增序号，循环使用 |
| 随机字符           | 2  | 防止并发重复          |

**生成方式：**

- 使用 Redis INCR 实现毫秒内序号
- Key: `task_no:sequence:{yyyyMMddHHmmss}`
- 过期时间: 1秒

### 2.2 组编号 (group\_no) 生成规则

**格式：** `GRP{yyyyMMdd}{8位随机数}`

**示例：** `GRP20260512A3F2B5C1`

### 3.3 任务主表 (api\_task)

**职责说明：任务表记录任务的核心状态和执行信息**

```sql
-- 任务主表：存储任务的核心数据和执行信息
-- 每次API调用的详细记录都在 api_task_log 表中
CREATE TABLE api_task (
    id                              BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    task_no                         VARCHAR(64) NOT NULL COMMENT '任务编号',
    source                          VARCHAR(64) NOT NULL DEFAULT 'API' COMMENT '操作来源：API-接口调用，SCHEDULE-定时任务，MANUAL-手动触发',
    group_no                        VARCHAR(64) COMMENT '任务批次编号（用于批量任务管理）',
    api_code                        VARCHAR(64) NOT NULL COMMENT 'API编码',
    api_name                        VARCHAR(128) COMMENT 'API名称(冗余)',

    action_type                     VARCHAR(32) NOT NULL DEFAULT 'ASYNC' COMMENT '动作类型：SYNC-同步执行（阻塞等待结果），ASYNC-异步执行（提交后立即返回）',

    status                          VARCHAR(32) NOT NULL DEFAULT 'PENDING' COMMENT '任务状态：PENDING-待处理，RUNNING-执行中，SUCCESS-成功，FAILED-失败，CANCELED-已取消',

    interrupt_flag                  TINYINT NOT NULL DEFAULT 0 COMMENT '中断标志：0-未中断，1-已中断',
    compensate_status               VARCHAR(32) NOT NULL DEFAULT 'NONE' COMMENT '补偿状态：NONE-无需补偿，PENDING-待补偿，RUNNING-补偿中，SUCCESS-补偿成功，FAILED-补偿失败',
    compensate_retry_count          INT NOT NULL DEFAULT 0 COMMENT '补偿重试次数',
    compensate_next_retry_time_ms   BIGINT COMMENT '下次补偿重试时间戳(毫秒)',

    priority                        INT NOT NULL DEFAULT 10 COMMENT '优先级1-10，1最高',

    request_context                 JSON NOT NULL COMMENT '请求上下文JSON',
    exec_info                       JSON COMMENT '执行信息JSON(包含插件步骤列表)',
    response_data                   JSON COMMENT '响应数据JSON',
    expire_time_ms                  BIGINT COMMENT '任务过期时间戳(毫秒)，超过该时间任务将不再执行',

    receipt_config                  JSON COMMENT '回执配置JSON',
    receipt_info                    JSON COMMENT '回执执行记录JSON',

    retry_count                     INT NOT NULL DEFAULT 0 COMMENT '已重试次数',
    max_retry_count                 INT NOT NULL DEFAULT 64 COMMENT '最大重试次数',
    next_retry_time_ms              BIGINT COMMENT '下次重试时间戳(毫秒)',

    create_time_ms                  BIGINT NOT NULL COMMENT '创建时间戳(毫秒)',
    start_execute_time_ms           BIGINT COMMENT '开始执行时间戳(毫秒)',
    end_execute_time_ms             BIGINT COMMENT '结束执行时间戳(毫秒)',
    execute_duration_ms             BIGINT COMMENT '执行耗时(毫秒)',

    cancel_time_ms                  BIGINT COMMENT '取消时间戳(毫秒)',
    cancel_reason                   VARCHAR(512) COMMENT '取消原因',
    canceled_by                     VARCHAR(64) COMMENT '取消人',

    update_time_ms                  BIGINT NOT NULL COMMENT '更新时间戳(毫秒)',
    deleted                        TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0-未删除，1-已删除',
    version                         INT NOT NULL DEFAULT 0 COMMENT '版本号',

    PRIMARY KEY (id),
    UNIQUE KEY uk_api_task_no_deleted (task_no, deleted),
    INDEX idx_task_status_interrupt (status, interrupt_flag, priority, create_time_ms),
    INDEX idx_task_retry (status, next_retry_time_ms)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='任务主表';
```

| version                  | INT          | 是  | 0       | 乐观锁                                   |

**任务状态流转规则：**

| 当前状态 | 目标状态 | 触发条件 | 补偿状态变化 |
|---------|---------|---------|------------|
| PENDING | RUNNING | 执行线程拉取任务 | - |
| PENDING | CANCELED | 用户取消（未执行） | - |
| RUNNING | SUCCESS | 执行成功 | compensate_status=SUCCESS |
| RUNNING | FAILED | 执行异常 | compensate_status=PENDING（自动补偿） |
| RUNNING | CANCELED | 用户中断+补偿完成 | compensate_status=SUCCESS |
| FAILED | PENDING | 自动/手动重试 | compensate_status=NONE |

**补偿状态说明：**

| 补偿状态 | 说明 |
|---------|------|
| NONE | 无需补偿 |
| PENDING | 待补偿（等待XXL-Job扫描执行） |
| RUNNING | 补偿执行中 |
| SUCCESS | 补偿成功 |
| FAILED | 补偿失败（等待XXL-Job重试补偿） |

**中断机制说明：**
- 用户发起取消时，设置 interrupt_flag=1
- 执行线程收到中断信号后，等待当前步骤完成
- 然后执行补偿逻辑，补偿完成后状态变为 CANCELED

### 3.4 request\_context JSON结构

**请求上下文：调用方给我的所有请求数据**

```json
{
    "params": {
        "orderId": "AMZ123456789",
        "startDate": "2024-01-01",
        "endDate": "2024-01-31"
    },
    "customData": {
        "orderNo": "SO-2024-001",
        "customerId": "CUST001"
    }
}
```

**字段说明：**

| 字段         | 类型     | 必填 | 说明                                                 |
| ---------- | ------ | -- | -------------------------------------------------- |
| params     | Object | 是  | **核心请求参数**：handler根据apiCode找到对应的处理逻辑，使用这些参数执行API调用 |
| customData | Object | 否  | **自定义数据**：原样返回给调用方，用于业务关联                          |

**重要设计说明：**

- 调用方只需要传递 `params` 和 `customData`
- handler 根据 `apiCode` 找到对应的处理逻辑，使用这些参数执行API调用
- 调用方不需要知道请求URL是什么、要不要加密、要不要签名，这些都是handler内部处理的

### 3.5 exec\_info 执行信息结构

**执行信息：记录任务执行过程中各插件的执行步骤，用于补偿和审计**

```json
{
    "steps": [
        {
            "stepOrder": 1,
            "pluginCode": "PARAM_VALIDATOR",
            "pluginName": "参数校验插件",
            "status": "SUCCESS",
            "inputData": {...},
            "outputData": {...},
            "executeStartTimeMs": 1746153600001,
            "executeEndTimeMs": 1746153600050,
            "executeCostTimeMs": 49,
            "compensateData": null
        },
        {
            "stepOrder": 2,
            "pluginCode": "RATE_LIMIT_CHECK",
            "pluginName": "限流检查插件",
            "status": "SUCCESS",
            "inputData": {...},
            "outputData": {...},
            "executeStartTimeMs": 1746153600051,
            "executeEndTimeMs": 1746153600100,
            "executeCostTimeMs": 49,
            "compensateData": null
        }
    ],
    "totalCostTimeMs": 149,
    "retryCount": 0
}
```

**字段说明：**

| 字段                       | 类型      | 说明                                           |
| ------------------------ | ------- | -------------------------------------------- |
| steps                    | Array   | 插件执行步骤列表，按执行顺序排列                        |
| steps[].stepOrder        | Integer | 步骤序号                                       |
| steps[].pluginCode       | String  | 插件编码                                       |
| steps[].pluginName       | String  | 插件名称                                       |
| steps[].status           | String  | 步骤状态：SUCCESS/FAILED                      |
| steps[].inputData        | Object  | 步骤输入数据                                    |
| steps[].outputData       | Object  | 步骤输出数据                                    |
| steps[].executeStartTimeMs | Long  | 步骤开始执行时间戳(毫秒)                          |
| steps[].executeEndTimeMs | Long    | 步骤结束执行时间戳(毫秒)                          |
| steps[].executeCostTimeMs | Long    | 步骤执行耗时(毫秒)                              |
| steps[].compensateData   | Object  | 补偿数据：补偿时从此字段获取补偿所需信息                 |
| totalCostTimeMs          | Long    | 任务总执行耗时(毫秒)                              |
| retryCount               | Integer | 重试次数                                       |

**补偿数据说明：**

- compensateData 由各插件在执行时设置，用于补偿时恢复状态
- 补偿时按 steps 倒序执行，即最后执行的插件先补偿
- 如果 compensateData 为 null，表示该步骤无需补偿

### 3.6 任务日志表 (api\_task\_log)

**职责说明：记录任务的每一次操作、每一次API调用，是问题排查和业务审计的核心**

**重要：一个任务可能涉及多次API调用（如查询订单列表后再查询详情），每次调用都要记录**

```sql
-- 任务日志表：记录任务的完整生命周期和每次API调用的详细信息
-- 这是业务审计和问题排查的核心表
-- 日志内容全部存储在log_data JSON字段中
CREATE TABLE api_task_log (
    id                              BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    task_no                         VARCHAR(64) NOT NULL COMMENT '任务编号',

    log_type                        VARCHAR(64) NOT NULL COMMENT '日志类型',

    log_data                        JSON NOT NULL COMMENT '日志内容JSON',

    create_time_ms                  BIGINT NOT NULL COMMENT '创建时间戳(毫秒)',

    PRIMARY KEY (id),
    INDEX idx_task_log_task_no (task_no)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='任务日志表';
```

**字段统计：共5个字段**

| 字段               | 类型          | 必填 | 默认值 | 说明                                      |
| ---------------- | ----------- | -- | --- | --------------------------------------- |
| id               | BIGINT      | 是  | 自增  | 主键                                      |
| task\_no         | VARCHAR(64) | 是  | -   | 任务编号                                    |
| log\_type        | VARCHAR(64) | 是  | -   | 日志类型                                    |
| log\_data        | JSON        | 是  | -   | **日志内容JSON**：包含请求参数、响应结果、错误信息、调用深度等所有信息 |
| create\_time\_ms | BIGINT      | 是  | -   | 创建时间戳(毫秒)                               |

**日志类型说明（log\_type）：**

## 8. JSON字段结构

### 8.1 rate_limit_config 限流规则配置

**配置说明：支持多维度、多规则的灵活限流配置**

```json
{
    "enabled": true,
    "rules": [
        {
            "name": "账号维度限流",
            "type": "CONCURRENCY",
            "dimension": "ACCOUNT",
            "keyTemplate": "${params.accountId}",
            "limit": 100,
            "windowSeconds": 1
        },
        {
            "name": "API维度限流",
            "type": "QPS",
            "dimension": "API",
            "keyTemplate": "${apiCode}",
            "limit": 50,
            "windowSeconds": 1
        },
        {
            "name": "自定义维度限流",
            "type": "CONCURRENCY",
            "dimension": "CUSTOM",
            "keyTemplate": "${params.accountId}:${params.region}",
            "limit": 200,
            "windowSeconds": 1
        }
    ]
}
```

**字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| enabled | Boolean | 是否启用限流 |
| rules | Array | 限流规则列表，支持多条规则 |
| rules[].name | String | 规则名称 |
| rules[].type | String | 限流类型：CONCURRENCY-并发数，QPS-每秒请求数 |
| rules[].dimension | String | 限流维度：ACCOUNT-账号维度，API-API维度，CUSTOM-自定义维度 |
| rules[].keyTemplate | String | 限流Key模板，支持变量替换 |
| rules[].limit | Integer | 限流阈值 |
| rules[].windowSeconds | Integer | 限流窗口时间(秒) |

**维度说明：**

| 维度 | keyTemplate示例 | 说明 |
|------|----------------|------|
| ACCOUNT | `${params.accountId}` | 根据请求参数中的账号ID限流 |
| API | `${externalApiCode}` | 根据外部API编码限流 |
| CUSTOM | `${params.accountId}:${params.region}` | 根据自定义拼接Key限流 |

**内置变量：**
- `${apiCode}` - API配置编码
- `${externalApiCode}` - 外部API编码
- `${params.xxx}` - 请求参数中的任意字段
- `${source}` - 任务来源

## 11. Redis数据结构

### 11.1 Key设计

| Key模式 | 类型 | 说明 | 过期时间 |
|---------|------|------|---------|
| rate_limit:{dimension}:{key} | String | 限流计数器（根据配置动态生成Key） | 窗口时间 |
| task_lock:{taskNo} | String | 任务执行锁 | 5分钟 |
| config:api:{apiCode} | Hash | API配置缓存 | 5分钟 |

**限流Key生成规则：**
- Key格式：`rate_limit:{dimension}:{expandedKey}`
- dimension：限流维度（account/api/custom）
- expandedKey：根据keyTemplate变量替换后的实际Key

**示例：**
- `rate_limit:account:AMZ123` - 账号维度限流
- `rate_limit:api:createOrder` - API维度限流
- `rate_limit:custom:AMZ123:us-west-2` - 自定义维度限流

### 7.2 限流实现方式

| 限流类型 | Redis操作 | 说明 |
|----------|-----------|------|
| CONCURRENCY | INCR/DECR | 并发数控制 |
| QPS | ZSET滑动窗口 | 每秒请求数控制 |

### 7.3 Redis操作示例

```bash
# ========== 并发限流（CONCURRENCY）==========
# 根据keyTemplate生成实际Key后执行
# 例如：keyTemplate="${params.accountId}"，实际参数accountId=AMZ123
# 实际Key = rate_limit:account:AMZ123

# 任务开始执行时：增加并发计数
INCR rate_limit:account:AMZ123
# 检查是否超过限制
GET rate_limit:account:AMZ123
# 任务执行完成时：减少并发计数
DECR rate_limit:account:AMZ123

# ========== QPS限流（滑动窗口）==========
# 添加当前请求到窗口
ZADD rate_limit:api:createOrder <current_timestamp_ms> <taskNo>
# 移除窗口外的过期请求（窗口1秒）
ZREMRANGEBYSCORE rate_limit:api:createOrder 0 <current_timestamp_ms - 1000>
# 统计当前窗口内的请求数量
ZCARD rate_limit:api:createOrder

# ========== 任务锁 ==========
SET task_lock:TASK2026050200001 "locked" NX EX 300
DEL task_lock:TASK2026050200001

# ========== API配置缓存 ==========
HSET config:api:AMAZON_ORDER_QUERY status "ENABLED" rateLimitConfig '{...}'
HGETALL config:api:AMAZON_ORDER_QUERY
```

## 12. 性能优化建议

### 12.1 索引优化

- **联合索引顺序**：遵循最左前缀原则，区分度高的字段放前面
- **索引选择性**：优先为区分度高的字段创建索引（如api\_code）
- **避免过度索引**：索引会增加写操作开销，仅为常用查询创建索引
- **覆盖索引**：对于频繁查询，考虑创建包含所有查询字段的覆盖索引

### 12.2 查询优化

- **分页查询**：所有列表查询必须使用分页，避免一次性返回大量数据
- **日期范围查询**：使用 create\_time\_ms 进行范围查询时，结合 status 或 api\_code 等条件缩小范围
- **避免全表扫描**：确保所有查询都使用索引，可以通过 EXPLAIN 分析执行计划
- **批量操作**：任务批量提交时，使用批量插入减少数据库交互

### 12.3 数据归档

- **历史数据归档**：定期将已完成的历史任务（如超过 30 天）归档到历史表
- **日志清理**：任务日志表定期清理（如保留 90 天），避免表过大影响查询性能
- **分批删除**：删除大量数据时使用分批删除，避免锁表和长事务

### 12.4 架构优化

- **读写分离**：主库处理写操作，从库处理读操作（尤其是任务查询和日志查询）
- **分库分表**：当数据量超过千万级时，考虑按 api\_code 或时间进行分表
- **缓存策略**：利用 Redis 缓存热点数据，减少数据库压力
