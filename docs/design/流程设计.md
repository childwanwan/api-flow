## 1. 核心流程设计

### 1.1 状态定义

| 状态代码 | 状态名称 | 说明 | 可转换状态 |
|---------|---------|------|-----------|
| PENDING | 待执行 | 任务已创建，等待扫描调度 | RUNNING, CANCELED |
| RUNNING | 执行中 | 任务正在执行 | SUCCESS, FAILED, CANCELED |
| SUCCESS | 执行成功 | 任务执行成功 | - |
| FAILED | 执行失败 | 任务执行失败 | PENDING(重试) |
| CANCELED | 已取消 | 任务被取消 | - |

### 1.2 补偿状态定义（独立字段）

| 补偿状态 | 说明 |
|---------|------|
| NONE | 无需补偿 |
| PENDING | 待补偿 |
| RUNNING | 补偿执行中 |
| SUCCESS | 补偿成功 |
| FAILED | 补偿失败 |

### 1.3 状态流转规则

| 当前状态 | 目标状态 | 触发条件 | 补偿动作 |
|---------|---------|---------|---------|
| PENDING | RUNNING | 扫描器发现任务，执行 | - |
| PENDING | CANCELED | 用户取消 | - |
| RUNNING | SUCCESS | 执行成功 | - |
| RUNNING | FAILED | 执行失败 | 根据插件的compensateData执行补偿 |
| RUNNING | CANCELED | 用户取消+补偿完成 | 按步骤倒序执行补偿 |
| FAILED | PENDING | 重试操作，重置状态 | - |

### 2.2 命令端设计要点

| 设计要点 | 实现方式 | 说明 |
|---------|---------|------|
| 任务持久化 | MySQL | 完全可控，可随时查询、修改、中断 |
| 领域事件通知 | Kafka | 轻量级通知，用于唤醒执行线程 |
| 任务存储 | MySQL | 队列只是通知机制，不是任务存储 |
| 立即执行 | SYNC模式 | 可直接执行，不走队列 |

### 2.3 领域事件设计

**Kafka Topic配置：**

| Topic | 说明 | 分区数 | 副本数 | 消费模式 |
|-------|------|---------|---------|---------|
| api_task_submitted | 新任务提交事件 | 16 | 3 | 集群消费 |
| api_task_retry | 任务重试事件 | 16 | 3 | 集群消费 |
| api_task_interrupt | 任务中断广播事件 | 1 | 3 | 广播模式（每个节点独立消费组） |

**领域事件消息结构：**

**TaskSubmittedEvent事件：**
```json
{
    "eventType": "TASK_SUBMITTED",
    "taskNo": "TASK2026050212345612345A3F",
    "apiCode": "AMAZON_ORDER_QUERY",
    "actionType": "ASYNC",
    "groupNo": "GRP20260502A1B2C3D4",
    "createTimeMs": 1748800000000
}
```

**TaskRetryEvent事件：**
```json
{
    "eventType": "TASK_RETRY",
    "taskNo": "TASK2026050212345612345A3F",
   apiCode": "AMAZON_ORDER_QUERY",
    "retryCount": 1,
    "nextExecuteTimeMs": 1748800060000,
    "createTimeMs": 1748800000000
}
```

**TaskInterruptEvent事件：**
```json
{
    "eventType": "TASK_INTERRUPT",
    "taskNo": "TASK2026050212345612345A3F",
    "interruptTimeMs": 1748800000000,
    "reason": "用户取消"
}
```

**Kafka广播消费配置：**
```
节点1：group.id = interrupt-consumer-node1
节点2：group.id = interrupt-consumer-node2
节点3：group.id = interrupt-consumer-node3
→ 每个节点都能收到同一条消息（广播模式）
```

### 2.4 请求参数结构

**核心设计思路：**
- taskNo: 任务编号，全局唯一
- apiCode: API编码，对应配置
- actionType: 执行类型，SYNC/ASYNC
- source: 来源，API/SCHEDULE/MANUAL
- groupNo: 任务组编号，用于批量管理
- priority: 优先级，1-10（1最高）
- requestContext: 请求上下文，包含params和customData
- receiptConfig: 回执配置（可选）

### 2.5 任务编号规则

**格式：** `TASK{yyyyMMddHHmmss}{毫秒内序号}{随机字符}`

**示例：** `TASK2026050212345612345A3F`

生成规则说明：
- 使用Redis维护毫秒内序号，确保全局唯一
- 序号自动在1秒后过期
- 随机字符增加安全性

### 2.6 响应参数结构

SYNC模式立即返回：
- taskNo: 任务编号
- status: 状态
- responseData: 响应数据
- executeTime: 执行耗时

ASYNC模式立即返回：
- taskNo: 任务编号
- status: 状态（PENDING）

## 3. 任务执行流程

### 3.1 PENDING任务扫描与执行流程

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          PENDING任务扫描与执行流程                               │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                │
│    线程从休眠中被领域事件唤醒                                                    │
│         │                                                                      │
│         ▼                                                                      │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │                    1. 从MySQL拉取待执行任务                               │ │
│  │   SELECT * FROM api_task                                                 │ │
│  │   WHERE status = 'PENDING' AND interrupt_flag = 0                       │ │
│  │   ORDER BY priority ASC, create_time_ms ASC                             │ │
│  │   LIMIT 100（批量拉取，可配置）                                           │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    │                                          │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │                    2. 遍历待执行任务                                      │ │
│  │   FOR EACH task IN tasks                                                 │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    │                                          │
│                                    ▼                                          │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │                    3. 获取Redis分布式锁                                  │ │
│  │   - SET task_lock:{taskNo} "locked" NX EX 300                           │ │
│  │   - 锁过期时间：300秒（5分钟，可配置）                                   │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    │                                          │
│                         ┌──────────┴──────────┐                               │
│                         ▼                       ▼                               │
│  ┌─────────────────────────────────┐  ┌─────────────────────────────────┐   │
│  │        4a. 获取锁成功            │  │        4b. 获取锁失败           │   │
│  │   - 更新任务状态为RUNNING（乐观锁） │ │  │   - 其他实例已处理，跳过       │ │
│  │   - startExecuteTimeMs = now     │ │  │                                 │ │
│  └─────────────────────────────────┘  └─────────────────────────────────┘   │
│                                    │                                          │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │                    5. 检查中断标记                                        │ │
│  │   - 检查本地中断标记                                                     │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    │                                          │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │                    6. 构建插件责任链                                      │ │
│  │   - 根据apiCode获取API实现类                                             │ │
│  │   - 如果配置了executorClass，使用配置的                                 │ │
│  │   - 否则用apiCode自动匹配@Plugin注解实现类                               │ │
│  │   - 加载所有启用的插件（内置+API执行插件）                               │ │
│  │   - 按order排序组成责任链                                                │ │
│  │   - 创建PluginContext上下文                                             │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    │                                          │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │                    7. 按插件责任链顺序执行                                │ │
│  │   - 在每个插件执行前检查中断标志                                         │ │
│  │   - 每个插件记录执行日志到exec_info.steps                               │ │
│  │   - 插件可以中断链式调用                                                 │ │
│  │   - API执行插件执行核心业务逻辑                                         │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    │                                          │
│                         ┌──────────┴──────────┐                               │
│                         ▼                       ▼                               │
│  ┌─────────────────────────────────┐  ┌─────────────────────────────────┐   │
│  │        8a. 执行成功              │  │        8b. 执行失败              │   │
│  │   - 更新任务结果和状态（SUCCESS） │ │  │   - 判断是否可重试             │ │
│  │   - endExecuteTimeMs = now       │ │  │   - 更新任务状态为FAILED       │ │

│  └─────────────────────────────────┘  └─────────────────────────────────┘   │
│                    │                             │                          │
│                    │                             ▼                          │
│                    │                  ┌─────────────────────────────────┐   │
│                    │                  │        8b.1 可重试             │   │

│                    │                             │                          │

│                                     ▼                                          │

│  │                    9. 释放Redis分布式锁                                    │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    │                                          │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │                    10. 执行回执通知                                       │ │
│  │   - 根据receipt_config执行HTTP/MQ回执                                   │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                                                                │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 执行端设计要点

| 设计要点 | 实现方式 | 说明 |
|---------|---------|------|
| 任务存储 | MySQL | 从数据库拉取，完全可控 |
| 批量拉取 | LIMIT 100 | 一次拉取多个任务，减少数据库交互 |
| 休眠机制 | 领域事件唤醒 | 无任务时线程休眠，收到事件后唤醒 |
| 分布式锁 | Redis SETNX | 防止多实例重复执行同一任务 |
| 责任链模式 | 插件按顺序执行 | API实现也是插件的一部分 |
| 步骤级日志 | Kafka异步记录 | 每个插件执行都记录，不影响主线程 |

## 4. 限流流程

### 4.1 限流类型

| 限流类型 | 说明 | 实现方式 |
|---------|------|---------|
| CONCURRENCY | 并发数限制 | Redis计数器 |
| QPS | 每秒请求数 | 滑动窗口算法 |

## 9. 重试与补偿的边界定义

### 9.1 何时走重试

| 场景 | 条件 | 结果 |
|------|------|------|
| 任务执行失败 | retry_count < max_retry_count | 状态变为PENDING，延迟重试 |
| 任务执行失败 | retry_count >= max_retry_count | 状态保持FAILED，不再重试 |

### 9.2 何时走补偿

| 场景 | 条件 | 结果 |
|------|------|------|
| 用户取消 | 任务在执行中 | 等待当前步骤完成后执行补偿，状态变为CANCELED |
| 中断触发 | 任务在执行中 | 等待当前步骤完成后执行补偿，状态变为CANCELED |

### 9.3 重试与补偿的优先级

| 场景 | 处理顺序 | 说明 |
|------|---------|------|
| 执行中任务被取消 | 先完成当前步骤，再执行补偿 | 不进行重试 |
| 执行失败且未取消 | 先重试，重试失败后若需补偿则执行补偿 | 重试优先 |
| 补偿失败 | 可重试补偿，最多64次 | 补偿失败后状态保持FAILED |

## 13. 重试与退避策略

### 13.1 任务重试退避策略

| 重试次数 | 延迟时间 | 计算公式 |
|---------|---------|---------|
| 第1次 | 5秒 | base_interval * 2^0 |
| 第2次 | 10秒 | base_interval * 2^1 |
| 第3次 | 20秒 | base_interval * 2^2 |

其中任务重试base_interval默认5000ms（可配置）

### 13.2 补偿重试退避策略

| 重试次数 | 延迟时间 | 计算公式 |
|---------|---------|---------|
| 第1次 | 1秒 | base_interval * 2^0 |
| 第2次 | 2秒 | base_interval * 2^1 |
| 第3次 | 4秒 | base_interval * 2^2 |

其中补偿重试base_interval默认1000ms（可配置）
