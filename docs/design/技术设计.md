## 1. 技术选型

根据需求文档中"统一入口、流量控制、统一监控、统一回执、插件扩展"的业务目标，以及"高可用、高性能、可扩展"的质量目标，技术选型如下：

| 组件 | 技术选型 | 版本要求 | 选型理由 |
|------|---------|---------|----------|
| 基础框架 | Spring Boot | 4.0.6 | 社区成熟，生态完善，便于快速构建RESTful服务 |
| Java版本 | JDK | 25 | LTS版本，性能稳定，支持虚拟线程等新特性 |
| 数据库 | MySQL | 8.0+ | 关系型数据库，支持事务，适合存储任务数据，满足数据持久性>99.999%的需求 |
| ORM | MyBatis-Plus | 3.5.x | 简化CRUD操作，支持分页、乐观锁等特性 |
| 缓存/限流 | Redis | 7.x | 高性能内存数据库，适合限流计数、分布式锁、任务编号生成 |
| 消息队列 | 抽象层+实现 | 可插拔 | 消息队列抽象层，支持Kafka/RocketMQ/RabbitMQ等多种实现，默认支持广播消费模式，满足异步通知和日志记录需求 |
| 任务调度 | XXL-JOB | 2.4.x | 成熟的分布式任务调度框架，用于超时任务兜底处理 |
| 前端模板 | FreeMarker | 2.3.x | 轻量级模板引擎，xxl-job-admin风格，前后端一体，.ftl模板放在resources/templates目录 |
| 构建工具 | Maven | 3.9+ | 依赖管理和项目构建 |

### 3.2 错误码枚举定义

根据需求中"错误码规范需求"的功能需求，定义完整错误码枚举：

```java
// common/exception/enums/ErrorCode.java
public enum ErrorCode {

    // API相关错误 (10001-10002)
    API_NOT_FOUND("10001", "API不存在"),
    API_DISABLED("10002", "API已禁用"),

    // 请求参数错误 (10003-10005)
    PARAM_VALIDATION_FAILED("10003", "参数校验失败"),
    RATE_LIMIT_TRIGGERED("10004", "限流触发"),
    SYSTEM_BUSY("10005", "系统繁忙"),

    // 任务相关错误 (10006-10007)
    TASK_NOT_FOUND("10006", "任务不存在"),
    TASK_STATUS_NOT_ALLOWED("10007", "任务状态不允许此操作"),

    // 插件相关错误 (10008, 10011)
    PLUGIN_NOT_FOUND("10008", "插件不存在"),
    PLUGIN_EXECUTE_FAILED("10011", "插件执行失败"),
    PLUGIN_IN_USE("10014", "插件正在使用中，无法卸载"),

    // 执行相关错误 (10009-10010)
    EXECUTE_TIMEOUT("10009", "执行超时"),
    EXECUTE_FAILED("10010", "执行失败"),

    // 补偿相关错误 (10012-10013)
    COMPENSATE_TIMEOUT("10012", "补偿超时"),
    COMPENSATE_FAILED("10013", "补偿失败"),

    // 未知错误
    UNKNOWN_ERROR("99999", "未知错误");
    
    private final String code;
    private final String message;
    
    ErrorCode(String code, String message) {
        this.code = code;
        this.message = message;
    }
    
    public String getCode() {
        return code;
    }
    
    public String getMessage() {
        return message;
    }
}
```

### 3.4 DDD分层依赖关系

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           DDD分层依赖关系                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│                              interfaces                                         │
│                           (依赖 application)                                   │
│                              │                                                 │
│                              ▼                                                 │
│                           application                                          │
│                    (依赖 domain， infrastructure)                              │
│                    ┌────────────┬────────────┐                               │
│                    ▼            ▼            ▼                                │
│               domain      infrastructure    common                             │
│           (核心业务逻辑)    (技术实现)                                         │
│               ▲            ▲                                                  │
│               │            │                                                  │
│               └────────────┘                                                  │
│            domain 不依赖 infrastructure                                        │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**设计原则说明**：
- **接口层(interfaces)**：负责接收HTTP请求和返回响应，不包含业务逻辑
- **应用层(application)**：编排领域服务，实现用例流程，不包含核心业务逻辑
- **领域层(domain)**：包含核心业务逻辑、领域模型、领域服务，不依赖基础设施
- **基础设施层(infrastructure)**：技术实现细节，如数据库访问、消息队列、定时任务等
- **公共模块(common)**：通用工具类、常量、异常定义

```java
public TaskSubmitResponse submitTask(TaskSubmitRequest request) {

    validateRequest(request);
    
    // 2. 查询API配置
    ApiConfigEntity config = apiConfigRepository.findByApiCode(request.getApiCode());

    }

    if (!rateLimitResult.isAllowed()) {

    }
    
    // 5. 创建任务实体
    TaskEntity task = TaskEntity.create(request, config);
    
    // 6. 持久化任务
    task = taskRepository.save(task);
    
    // 7. 根据actionType分发
    if (ActionType.SYNC.equals(task.getActionType())) {
        // SYNC模式：立即执行
        TaskResult result = apiExecutorService.execute(task);
        task.complete(result.getResponseData());
        taskRepository.update(task);

        domainEventPublisher.publish(new TaskSubmittedEvent(task.getTaskNo()));
        return TaskSubmitResponse.pending(task);
```

### 6.2 任务执行流程

根据需求中"任务执行流程"的业务流程需求，技术实现如下：

```java
// TaskExecuteService.executeTask() 方法流程
public void executeTask(String taskNo) {
    // 1. 查询任务
    TaskEntity task = taskRepository.findByTaskNo(taskNo);
    if (task == null || !task.canExecute()) {
        return;
    }

    // 2. 获取分布式锁 (SET task_lock:{taskNo} "locked" NX EX 300)
    String lockKey = "task_lock:" + taskNo;
    boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "locked", 300, TimeUnit.SECONDS);
    if (!locked) {
        return; // 其他节点正在执行
    }

    try {
        // 3. 更新状态为RUNNING（乐观锁）
        boolean updated = taskRepository.updateStatus(taskNo, TaskStatus.PENDING, TaskStatus.RUNNING);
        if (!updated) {
            return; // 状态已被修改
        }
        task.setStatus(TaskStatus.RUNNING);

        // 4. 检查中断标志（获取锁之后、执行之前检查）
        if (checkInterrupt(taskNo)) {
            handleInterrupt(task);
            return;
        }

        // 5. 构建插件责任链
        PluginChain chain = pluginFactory.buildChain(task);

        // 6. 执行责任链
        PluginResult result = chain.execute(buildPluginContext(task));

        // 7. 更新任务结果
        if (result.isSuccess()) {
            task.complete(result.getData());
        } else {
            task.fail(result.getErrorMessage());
        }

    } catch (TaskInterruptedException e) {
        handleInterrupt(task);
    } catch (Exception e) {
        task.fail(e.getMessage());
        log.error("任务执行失败: {}", taskNo, e);
    } finally {
        // 8. 释放分布式锁
        redisTemplate.delete(lockKey);
        // 9. 更新任务状态
        taskRepository.update(task);
        // 10. 执行回执通知（仅在任务成功完成时发送）
        if (TaskStatus.SUCCESS.equals(task.getStatus())) {
            sendReceipt(task);
        }
    }
}
```

**优化说明**：中断检查在获取锁之后、执行之前进行。这样设计的原因是：
1. 如果在获取锁之前检查中断，可能存在时序问题：检查时未中断，但获取锁期间任务被中断
2. 在获取锁之后检查，可以确保任务在确定执行前再次确认是否被中断
3. 如果发现中断，立即释放锁（通过handleInterrupt处理）并返回，避免无效执行

```java
public void executeCompensation(TaskEntity task, List<PluginStep> completedSteps) {

        // 更新补偿状态
        task.setCompensateStatus(CompensateStatus.PENDING);
        taskRepository.updateCompensateStatus(task.getTaskNo(), CompensateStatus.PENDING);
        
        task.setCompensateStatus(CompensateStatus.RUNNING);
        taskRepository.updateCompensateStatus(task.getTaskNo(), CompensateStatus.RUNNING);
        
        // 按倒序执行补偿
        for (int i = completedSteps.size() - 1; i >= 0; i--) {
            PluginStep step = completedSteps.get(i);
            Plugin plugin = pluginRegistry.getPlugin(step.getPluginCode());
            if (plugin != null) {
                plugin.compensate(buildPluginContext(task, step));
                logCompensation(task, step);
            }
        }

        task.setCompensateStatus(CompensateStatus.SUCCESS);
        task.setStatus(TaskStatus.CANCELED);
        taskRepository.update(task);
        
    } catch (Exception e) {
        // 补偿失败
        task.setCompensateStatus(CompensateStatus.FAILED);
        taskRepository.update(task);
        log.error("任务补偿执行失败: {}", task.getTaskNo(), e);
```

```java
        return task.getReceiptConfig();

        // 使用API级别的回执配置
```

## 11. 补偿超时机制设计

### 11.1 补偿超时控制

根据需求中"补偿超时时间"的功能需求，设计补偿超时机制：

```java
// 补偿超时配置
public class CompensateTimeoutConfig {
    // 补偿最大等待时间（秒），中断后等待当前步骤完成的最大时间
    public static final int COMPENSATE_MAX_WAIT_SECONDS = 30;
    // 强制中断超时时间（秒），超过此时间强制终止
    public static final int FORCE_INTERRUPT_SECONDS = 60;
    // 补偿最大重试次数
    public static final int MAX_COMPENSATE_RETRY_COUNT = 64;
}

// 补偿执行逻辑
public void executeCompensationWithTimeout(TaskEntity task, List<PluginStep> completedSteps) {
    long startTime = System.currentTimeMillis();
    
    try {
        task.setCompensateStatus(CompensateStatus.PENDING);
        taskRepository.updateCompensateStatus(task.getTaskNo(), CompensateStatus.PENDING);
        
        task.setCompensateStatus(CompensateStatus.RUNNING);
        taskRepository.updateCompensateStatus(task.getTaskNo(), CompensateStatus.RUNNING);
        
        for (int i = completedSteps.size() - 1; i >= 0; i--) {
            // 检查是否超过最大等待时间
            long elapsed = (System.currentTimeMillis() - startTime) / 1000;
            if (elapsed >= CompensateTimeoutConfig.COMPENSATE_MAX_WAIT_SECONDS) {
                log.warn("补偿等待超时，继续执行剩余补偿步骤");
            }
            
            PluginStep step = completedSteps.get(i);
            Plugin plugin = pluginRegistry.getPlugin(step.getPluginCode());
            if (plugin != null) {
                plugin.compensate(buildPluginContext(task, step));
                logCompensation(task, step);
            }
        }
        
        task.setCompensateStatus(CompensateStatus.SUCCESS);
        task.setStatus(TaskStatus.CANCELED);
        taskRepository.update(task);
        
    } catch (Exception e) {
        long elapsed = (System.currentTimeMillis() - startTime) / 1000;
        
        // 检查是否超过强制中断超时
        if (elapsed >= CompensateTimeoutConfig.FORCE_INTERRUPT_SECONDS) {
            log.error("补偿执行超时{}秒，强制终止: {}", CompensateTimeoutConfig.FORCE_INTERRUPT_SECONDS, task.getTaskNo());
            task.setCompensateStatus(CompensateStatus.FAILED);
            taskRepository.update(task);
            
            alarmService.send(AlarmEvent.builder()
                .eventType(AlarmEventTypes.COMPENSATION_TIMEOUT)
                .taskNo(task.getTaskNo())
                .build());
        } else {
            task.setCompensateStatus(CompensateStatus.FAILED);
            taskRepository.update(task);
        }
        log.error("任务补偿执行失败: {}", task.getTaskNo(), e);
        
        // 补偿失败后，如果重试次数小于最大限制，可以进行补偿重试
        // 具体重试逻辑由XXL-JOB兜底任务处理
    }
}
```

### 11.2 补偿重试机制

根据需求中"补偿失败重试，最多64次"的功能需求，设计补偿重试机制：

```java
// 补偿重试配置
public class CompensateRetryConfig {
    // 补偿最大重试次数
    public static final int MAX_COMPENSATE_RETRY_COUNT = 64;
    // 补偿重试基础间隔(毫秒)，采用指数退避: 1s, 2s, 4s, 8s...
    public static final long BASE_RETRY_INTERVAL_MS = 1000;
}

// 补偿重试退避计算
// 延迟时间 = base_interval * 2^(retryCount-1)
// 第1次重试: 1000 * 2^0 = 1000ms (1秒)
// 第2次重试: 1000 * 2^1 = 2000ms (2秒)
// 第3次重试: 1000 * 2^2 = 4000ms (4秒)
public class CompensateRetryUtil {

    public static long calculateCompensateRetryDelay(long baseIntervalMs, int retryCount) {
        return baseIntervalMs * (long) Math.pow(2, retryCount - 1);
    }

    public static void main(String[] args) {
        long baseInterval = CompensateRetryConfig.BASE_RETRY_INTERVAL_MS;
        for (int i = 1; i <= 5; i++) {
            System.out.println("第" + i + "次补偿重试延迟: " + calculateCompensateRetryDelay(baseInterval, i) + "ms");
        }
    }
}
```

#### 11.2.1 补偿重试字段设计

在api_task表中增加补偿重试相关字段：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| compensate_retry_count | INT | 当前补偿重试次数 |
| compensate_next_retry_time_ms | BIGINT | 下次补偿重试时间戳 |

#### 11.2.2 补偿重试流程

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           补偿重试流程                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  1. XXL-JOB每分钟扫描 compensate_status=FAILED 的任务                       │
│  2. 检查 compensate_retry_count < 64                                         │
│  3. 检查 compensate_next_retry_time_ms <= NOW()（到重试时间）               │
│  4. 计算下次重试时间 = now + base_interval * 2^(retryCount-1)                  │
│  5. 更新 compensate_retry_count++, compensate_next_retry_time_ms              │
│  6. 重置 compensate_status=PENDING，唤醒补偿执行                              │
│  7. 补偿成功则 compensate_status=SUCCESS，任务状态变为 CANCELED              │
│  8. 补偿失败则 compensate_retry_count++，继续重试流程                      │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

#### 11.2.3 补偿重试SQL示例

```sql
-- 扫描需要重试的补偿任务
SELECT * FROM api_task
WHERE compensate_status = 'FAILED'
  AND compensate_retry_count < 64
  AND compensate_next_retry_time_ms <= NOW()
ORDER BY compensate_next_retry_time_ms ASC
LIMIT 100;

-- 更新补偿重试次数和下次重试时间（乐观锁防止并发）
-- 重试延迟公式: base_interval(1000ms) * 2^(compensate_retry_count)
UPDATE api_task
SET compensate_retry_count = compensate_retry_count + 1,
    compensate_next_retry_time_ms = NOW() + 1000 * POWER(2, compensate_retry_count),
    compensate_status = 'PENDING'
WHERE task_no = #{taskNo}
  AND compensate_retry_count = #{expectedCount};
```

#### 11.2.4 补偿幂等性保证机制

补偿执行需要保证幂等性，防止重复执行：

| 机制 | 说明 |
|------|------|
| 乐观锁更新 | `compensate_retry_count`使用乐观锁，防止并发重复执行 |
| 步骤级补偿 | 每个插件的补偿操作具有幂等性（如删除只执行一次） |
| 执行记录 | `exec_info.steps`记录已执行步骤，补偿时跳过已完成的步骤 |
| 状态机控制 | PENDING → RUNNING → SUCCESS/FAILED |

#### 11.2.5 补偿执行幂等性实现

```java
// 补偿执行幂等性实现
public void executeCompensationWithIdempotency(TaskEntity task, List<PluginStep> completedSteps) {
    // 按步骤倒序执行补偿
    for (int i = completedSteps.size() - 1; i >= 0; i--) {
        PluginStep step = completedSteps.get(i);

        // 检查该步骤是否已补偿（通过compensateData判断）
        if (step.getCompensateData() == null) {
            continue; // 无需补偿的步骤跳过
        }

        // 使用乐观锁更新补偿状态
        int updated = taskRepository.updateCompensateStepStatus(
            task.getTaskNo(),
            step.getStepOrder(),
            "COMPENSATING"
        );

        if (updated == 0) {
            // 已被其他节点补偿，跳过
            continue;
        }

        try {
            // 执行补偿
            Plugin plugin = pluginRegistry.getPlugin(step.getPluginCode());
            plugin.compensate(buildPluginContext(task, step));

            // 补偿成功，清除compensateData
            step.setCompensateData(null);
            logCompensation(task, step);

        } catch (Exception e) {
            // 补偿失败，保留compensateData供下次重试
            step.setCompensateData(step.getCompensateData());
            log.error("步骤补偿失败: taskNo={}, step={}", task.getTaskNo(), step.getStepOrder(), e);
            throw e;
        }
    }
}
```

#### 12.2.2 回执重试幂等性设计

回执重试必须保证幂等性，防止重复发送：

```java
// 回执重试幂等性保证
public class ReceiptRetryExecutor {

    // 生成唯一回执请求ID，用于幂等校验
    private String generateReceiptRequestId(String taskNo, int retryCount) {
        return taskNo + "_receipt_" + retryCount;
    }

    public void executeReceiptRetry(TaskEntity task) {
        ReceiptInfo receiptInfo = parseReceiptInfo(task.getReceiptInfo());

        // 检查是否超过最大重试次数
        if (receiptInfo.getRetryCount() >= receiptInfo.getMaxRetryCount()) {
            receiptInfo.setStatus("FAILED");
            updateTaskReceiptInfo(task, receiptInfo);
            return;
        }

        // 检查是否到达重试时间
        if (System.currentTimeMillis() < receiptInfo.getNextRetryTimeMs()) {
            return; // 未到重试时间，跳过
        }

        try {
            // 增加重试计数（乐观锁，防止重复执行）
            int expectedCount = receiptInfo.getRetryCount();
            int updated = taskRepository.updateReceiptRetryCount(
                task.getTaskNo(),
                expectedCount,
                expectedCount + 1
            );

            if (updated == 0) {
                // 已经被其他线程更新，跳过
                return;
            }

            // 生成幂等请求ID
            String requestId = generateReceiptRequestId(
                task.getTaskNo(),
                receiptInfo.getRetryCount() + 1
            );

            // 发送回执（HTTP/MQ）
            boolean success = sendReceipt(task, receiptInfo, requestId);

            if (success) {
                receiptInfo.setStatus("SUCCESS");
                receiptInfo.setRetryCount(receiptInfo.getRetryCount() + 1);
            } else {
                handleReceiptFailure(receiptInfo);
            }

        } catch (Exception e) {
            handleReceiptFailure(receiptInfo);
        }

        updateTaskReceiptInfo(task, receiptInfo);
    }

    private boolean sendReceipt(TaskEntity task, ReceiptInfo info, String requestId) {
        if ("HTTP".equals(info.getReceiptType())) {
            HttpHeaders headers = new HttpHeaders();
            headers.add("X-Request-Id", requestId);
            headers.add("X-Task-No", task.getTaskNo());
            // 发送HTTP请求...
            return true;
        } else if ("MQ".equals(info.getReceiptType())) {
            // MQ消息携带requestId
            return true;
        }
        return false;
    }
}
```

### 14.3 任务编号生成算法

根据需求中"任务编号规则"的功能需求，设计任务编号生成算法：

```java
// 任务编号格式：TASK{yyyyMMddHHmmss}{5位序号}{3位随机字符}
// 示例：TASK2026050212345612345A3F
// 安全性：同一毫秒内支持最多 36^3 = 46656 个不重复随机字符组合
//         序号支持 0-99999，共10万个序号
//         每毫秒总共支持 10万 * 46656 = 46.6亿 个不重复编号
public class TaskNoGenerator {

    // 每毫秒内的序号（理论上支持每毫秒10万个请求）
    private static final AtomicLong seq = new AtomicLong(0);
    // 当前使用到的毫秒时间戳，用于判断是否需要重置序号
    private static long lastMillis = 0;

    private static final char[] RANDOM_CHARS = "ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789".toCharArray();

    public static synchronized String generate() {
        long currentMillis = System.currentTimeMillis();

        // 如果时间戳变了，重置序号计数器
        if (currentMillis != lastMillis) {
            lastMillis = currentMillis;
            seq.set(0);
        }

        String timestamp = new SimpleDateFormat("yyyyMMddHHmmss").format(new Date());
        long sequence = seq.incrementAndGet() % 100000;
        String random = generateRandom(3);

        return String.format("TASK%s%s%s", timestamp, String.format("%05d", sequence), random);
    }

    private static String generateRandom(int length) {
        StringBuilder sb = new StringBuilder(length);
        for (int i = 0; i < length; i++) {
            int index = (int) (Math.random() * RANDOM_CHARS.length);
            sb.append(RANDOM_CHARS[index]);
        }
        return sb.toString();
    }

    // 测试并发场景
    public static void main(String[] args) {
        Set<String> taskNos = ConcurrentHashMap.newKeySet();
        IntStream.range(0, 10000).parallel().forEach(i -> {
            taskNos.add(generate());
        });
        System.out.println("生成数量: " + taskNos.size() + ", 预期: 10000");
        System.out.println("示例: " + taskNos.stream().findAny().orElse("N/A"));
    }
}
```

**算法说明**：
- **序号部分**：5位数字（0-99999），每毫秒最多支持10万个不重复序号
- **随机字符**：3位字符（36^3 = 46656种组合），增加唯一性保障
- **同步机制**：使用synchronized保证同一毫秒内序号严格递增
- **时间戳变化处理**：当时间戳变化时自动重置序号计数器

**分布式环境说明**：
synchronized只保证单机并发安全。在多实例部署时，每个实例可能有独立的JVM内存，建议使用Redis INCR命令生成序号作为分布式补充方案：

```java
// Redis分布式序号生成
public String generateWithRedis() {
    String dateStr = new SimpleDateFormat("yyyyMMddHHmmss").format(new Date());
    String key = "task:seq:" + dateStr;
    Long seq = redisTemplate.opsForValue().increment(key);
    // 设置1秒过期，防止数据堆积
    redisTemplate.expire(key, 1, TimeUnit.SECONDS);
    String random = generateRandom(3);
    return String.format("TASK%s%s%s", dateStr, String.format("%05d", seq % 100000), random);
}
```

**并发安全性**：
- 单机并发：synchronized保证严格递增
- 分布式场景：建议在数据库层面添加唯一索引约束，防止极端情况下的重复
