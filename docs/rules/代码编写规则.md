## 1. DDD分层架构

### 1.1 分层原则

| 层级 | 职责 | 依赖关系 |
|------|------|----------|
| interfaces（接口层） | 接收HTTP请求，参数校验，响应处理 | 依赖application |
| application（应用层） | 编排业务用例，事务管理 | 依赖domain |
| domain（领域层） | 核心业务逻辑，状态流转，领域事件 | 不依赖外部 |
| infrastructure（基础设施层） | 技术实现（DB、Redis、MQ） | 实现domain接口 |

### 1.2 依赖规则

- **domain层禁止**使用@Service、@Component等Spring注解
- **domain层禁止**直接使用MyBatis、Redis、Kafka等客户端
- **domain层**只定义接口，由infrastructure实现
- **禁止循环依赖**

### 1.3 包结构规范

```
com.apigateway
├── interfaces              # 接口层
│   ├── {entity}/controller # 控制器
│   └── {entity}/dto       # DTO（request/response）
├── application            # 应用层
│   └── {entity}/service  # 应用服务
├── domain                 # 领域层
│   ├── {entity}/model    # 领域模型
│   ├── {entity}/service  # 领域服务
│   ├── {entity}/event    # 领域事件
│   └── {entity}/repository # 仓储接口
├── infrastructure         # 基础设施层
│   ├── persistence/mybatis # MyBatis实现
│   ├── persistence/redis  # Redis实现
│   └── messaging          # 消息队列实现
└── common                 # 公共模块
```

## 2. 数据库设计规范

### 2.1 字段命名规范

| 类型 | 命名规范 | 示例 |
|------|----------|------|
| 主键 | id | id |
| 业务ID | xxx_id | task_no, api_code |
| 时间戳 | xxx_time_ms | create_time_ms, start_execute_time_ms |
| 逻辑删除 | deleted | deleted (0-未删除, 1-已删除) |
| 版本号 | version | version |
| JSON字段 | xxx_config / xxx_info | rate_limit_config, exec_info |

### 2.2 字段类型规范

| 字段类型 | MySQL类型 | 说明 |
|----------|-----------|------|
| 主键 | BIGINT | 自增主键 |
| 业务ID | VARCHAR(64) | 业务编号 |
| 状态 | VARCHAR(32) | 状态枚举 |
| 时间 | BIGINT | 毫秒级时间戳 |
| JSON数据 | JSON | JSON格式 |
| 布尔 | TINYINT(1) | 0/1 |

### 2.3 时间字段规范

**所有时间字段使用BIGINT存储毫秒级时间戳，字段名以`_ms`结尾**

| 字段 | 说明 |
|------|------|
| create_time_ms | 创建时间戳 |
| update_time_ms | 更新时间戳 |
| start_execute_time_ms | 开始执行时间戳 |
| end_execute_time_ms | 结束执行时间戳 |
| cancel_time_ms | 取消时间戳 |

### 2.4 JSON字段处理规范

**数据库JSON字段用String存储，Java层使用强类型对象**

- DO层：JSON字段定义为String
- Entity层：使用强类型对象（禁止使用Map/HashMap/JSONObject存储JSON数据）
- Converter层：负责String与强类型对象之间的转换

**例外**：PluginContext中的params、customData、compensateData使用Map<String, Object>，因为这些是插件间的动态数据传递，必须具备灵活性

## 4. Redis使用规范

### 4.1 Key命名规范

| 类型 | 命名规范 | 示例 |
|------|----------|------|
| 分布式锁 | 小写+冒号分隔 | task_lock:{taskNo} |
| 限流计数 | 小写+冒号分隔 | rate:limit:{apiCode}:{dimension} |
| 序列号 | 小写+冒号分隔 | task_no:sequence:{yyyyMMddHHmmss} |
| 缓存 | 小写+冒号分隔 | api:config:{apiCode} |

### 4.2 分布式锁规范

**锁Key格式**：`{业务前缀}:lock:{业务标识}`

**示例**：
- 任务锁：`task_lock:{taskNo}`
- API配置锁：`config_lock:{apiCode}`

**锁参数**：
- 过期时间：300秒（可配置）
- 获取超时：0（立即返回）

### 4.3 缓存规范

- 缓存键必须设置过期时间
- 敏感数据禁止缓存
- 缓存更新采用删除策略而非更新策略

## 6. 核心模式规范

### 6.1 任务状态机

| 状态 | 可转换状态 |
|------|-----------|
| PENDING | RUNNING, CANCELED |
| RUNNING | SUCCESS, FAILED, CANCELED |
| SUCCESS | - |
| FAILED | PENDING（重试） |
| CANCELED | - |

### 6.2 补偿状态机

| 状态 | 说明 |
|------|------|
| NONE | 无需补偿 |
| PENDING | 待补偿 |
| RUNNING | 补偿中 |
| SUCCESS | 补偿成功 |
| FAILED | 补偿失败 |

### 6.3 重试退避策略

| 重试次数 | 延迟时间 | 计算公式 |
|---------|---------|---------|
| 第1次 | 5秒 | base_interval × 2^0 |
| 第2次 | 10秒 | base_interval × 2^1 |
| 第3次 | 20秒 | base_interval × 2^2 |
| ... | ... | ... |

- 任务重试base_interval：5000ms（可配置）
- 补偿重试base_interval：1000ms（可配置）
- 回执重试base_interval：1000ms（可配置）
- 最大重试次数：64（可配置）

### 6.4 exec_info结构规范

**说明**：status字段存储枚举的name()值，即RUNNING/SUCCESS/FAILED/SKIPPED

```json

      "status": "RUNNING",

      "executeStartTimeMs": 1746153600001,
      "executeEndTimeMs": 1746153600050,
      "executeCostTimeMs": 49,
```

## 8. 插件开发规范

### 8.1 插件接口

```java
public interface ApiPlugin {
    String getPluginCode();
    String getPluginName();
    default int getOrder() { return 1000; }
    default boolean isEnabled() { return true; }
    PluginResult execute(PluginContext context, PluginChain chain);
    default void compensate(PluginContext context) {}
}
```

### 8.2 插件执行顺序

- 按order字段升序执行
- 数值越小越先执行
- 插件链末端的插件负责调用外部API

### 8.3 插件上下文

**说明**：params、customData、compensateData使用Map<String, Object>是唯一例外，因为这些是插件间的动态数据传递，必须具备灵活性

```java
public class PluginContext {
    private TaskEntity task;
    private ApiConfigEntity config;
    private Map<String, Object> params;           // 插件间动态数据传递
    private Map<String, Object> customData;       // 插件间动态数据传递
    private Map<String, Object> compensateData;   // 插件间动态数据传递
    private ExecInfo execInfo;
}
```

## 13. 测试规范

### 13.1 测试覆盖率要求

- **单元测试覆盖率要求：100%**
- 核心业务逻辑必须编写单元测试
- 所有public方法必须编写测试用例

### 13.2 测试框架

| 框架 | 用途 |
|------|------|
| JUnit 5 | 单元测试基础框架 |
| Mockito | Mock对象创建和验证 |
| AssertJ | 断言库 |

### 13.3 测试命名规范

| 类型 | 命名规范 | 示例 |
|------|----------|------|
| 测试类 | {被测类}Test | TaskApplicationServiceTest |
| 测试方法 | should{预期行为}When{条件} | shouldReturnTaskNoWhenSubmitSuccess |
| 测试用例 | given{前置条件}when{操作}then{预期结果} | givenTaskEntityWhenCancelThenStatusIsCanceled |

### 13.4 Mockito使用规范

```java
// Mock依赖
@ExtendWith(MockitoExtension.class)
class TaskApplicationServiceTest {

    @Mock
    private TaskRepository taskRepository;

    @Mock
    private ApiConfigRepository apiConfigRepository;

    @InjectMocks
    private TaskApplicationService taskApplicationService;

    @Test
    void shouldReturnTaskNoWhenSubmitSuccess() {
        // given
        TaskSubmitRequest request = new TaskSubmitRequest();
        request.setApiCode("TEST_API");
        when(apiConfigRepository.findByApiCode("TEST_API"))
            .thenReturn(createMockConfig());

        // when
        TaskSubmitResponse response = taskApplicationService.submitTask(request);

        // then
        assertThat(response.getData().getTaskNo()).isNotNull();
        verify(taskRepository, times(1)).save(any(TaskEntity.class));
    }
}
```

### 13.5 测试组织规范

- 测试类与被测类同包名下，目录为test/java
- 每个测试方法必须包含given/when/then三部分
- 断言必须使用AssertJ风格
- Mock对象使用@Mock注解
- 注入使用@InjectMocks注解

### 13.6 测试质量标准

| 检查项 | 要求 |
|--------|------|
| 分支覆盖率 | 100% |
| 行覆盖率 | 100% |
| 方法覆盖率 | 100% |
| 循环语句 | 需设计独立测试用例覆盖循环边界 |
