## 1. 项目概述

### 1.1 项目背景

公司业务涉及多个外部平台的API对接，每个平台都有独立的API体系、限流规则、回执方式。为了统一管理这些API对内对外的服务，需要建设一个API接口网关中台。

### 1.2 业务目标

| 目标 | 说明 |
|------|------|
| 统一入口 | 所有API调用方通过统一入口接入 |
| 流量控制 | 智能限流，保护后端服务 |
| 统一监控 | 实时监控API调用情况和性能指标 |
| 统一回执 | 支持HTTP和MQ两种回执方式 |
| 插件扩展 | 支持插件化架构，针对特定API进行自定义处理 |

### 1.3 质量目标

| 目标 | 指标 |
|------|------|
| 高可用 | 保证99.9%以上的可用性 |
| 高性能 | 同步接口平均响应时间<500ms |
| 可扩展 | 支持水平扩展，应对流量高峰 |
| 可配置 | 所有规则可通过页面配置，无需修改代码 |
| 可观测 | 完善的日志和监控体系 |

## 3. 业务流程需求

### 3.1 任务提交流程

1. 调用方发起API请求
2. 系统校验请求参数
3. 检查API配置是否存在且启用
4. 校验过滤规则
5. 生成任务编号，创建任务记录（状态=PENDING）
6. 根据执行类型执行：
   - SYNC模式：立即执行，返回执行结果
   - ASYNC模式：发布TaskSubmittedEvent领域事件唤醒执行线程，返回任务编号

### 3.2 任务执行流程

1. 执行线程从MySQL数据库拉取待执行任务
   - 拉取条件：status=PENDING AND interrupt_flag=0
   - 拉取顺序：priority ASC, create_time_ms ASC
   - 拉取数量：默认每次拉取100条
2. 获取Redis分布式锁（SET task_lock:{taskNo} "locked" NX EX 300）
3. 更新任务状态为RUNNING（乐观锁）
4. 按插件责任链顺序执行
5. 记录每个步骤的执行日志到exec_info.steps
6. 更新任务结果和状态
7. 释放Redis分布式锁
8. 执行回执通知

### 3.3 任务中断流程

1. 用户发起取消/中断请求
2. 通过消息队列发布中断广播事件
3. 所有节点收到中断事件，检查本地任务缓存
4. 若该任务正在本节点执行，设置本地中断标志
5. 执行线程在每个插件执行前检查中断标志
6. 若收到中断信号，等待当前步骤完成后进入中断处理
7. 按步骤倒序执行补偿逻辑（从exec_info.steps获取已执行步骤列表）
8. 补偿完成后更新任务状态为CANCELED

### 3.4 补偿兜底流程

1. XXL-JOB定时任务每分钟扫描需要处理的任务
2. 扫描条件：
   - RUNNING状态超过配置的超时时间未完成（执行超时）
   - FAILED状态且retry_count < max_retry_count（可重试）
   - compensate_status=FAILED（补偿失败）
3. 处理策略：
   - 超时任务：检查执行线程是否存活，若断开则重置状态为PENDING
   - 失败任务：增加retry_count，使用指数退避计算下次执行时间，重置状态为PENDING
   - 补偿失败任务：重置compensate_status=PENDING，重新执行补偿逻辑

## 4. 补偿机制详细说明

### 4.1 补偿触发条件

| 触发场景 | 补偿逻辑 |
|---------|---------|
| 用户主动取消 | 执行已完成步骤的补偿 |
| 执行中断 | 执行已完成步骤的补偿 |
| 补偿失败重试 | 继续执行未完成的补偿 |

### 4.2 补偿数据来源

补偿数据来自任务执行过程中各插件保存的compensateData：
- 存储位置：PluginContext.compensateData
- 数据格式：Map<String, Object>
- 补偿时由各插件从PluginContext中获取自己的补偿数据

### 4.3 补偿超时时间

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 补偿最大等待时间 | 30秒 | 中断后等待当前步骤完成的最大时间 |
| 强制中断超时 | 60秒 | 超过此时间强制终止 |

### 4.4 步骤信息存储

步骤信息存储在任务的exec_info字段（JSON格式）：

```json
{
  "steps": [
    {
      "stepOrder": 1,
      "pluginCode": "PARAM_VALIDATOR",
      "pluginName": "参数校验插件",
      "status": "SUCCESS",
      "inputData": {...},
      "outputData": {...},
      "executeStartTimeMs": 1746153600001,
      "executeEndTimeMs": 1746153600050,
      "executeCostTimeMs": 49,
      "compensateData": null
    },
    {
      "stepOrder": 2,
      "pluginCode": "RATE_LIMIT_CHECK",
      "pluginName": "限流检查插件",
      "status": "SUCCESS",
      "inputData": {...},
      "outputData": {...},
      "executeStartTimeMs": 1746153600051,
      "executeEndTimeMs": 1746153600100,
      "executeCostTimeMs": 49,
      "compensateData": null
    }
  ],
  "totalCostTimeMs": 149,
  "retryCount": 0
}
```

## 5. 重试与补偿的边界定义

### 5.1 何时走重试

| 场景 | 条件 | 结果 |
|------|------|------|
| 任务执行失败 | retry_count < max_retry_count | 状态变为PENDING，延迟重试 |
| 任务执行失败 | retry_count >= max_retry_count | 状态保持FAILED，不再重试 |

### 5.2 何时走补偿

| 场景 | 条件 | 结果 |
|------|------|------|
| 用户取消 | 任务在执行中 | 等待当前步骤完成后执行补偿，状态变为CANCELED |
| 中断触发 | 任务在执行中 | 等待当前步骤完成后执行补偿，状态变为CANCELED |

### 5.3 重试与补偿的优先级

| 场景 | 处理顺序 | 说明 |
|------|---------|------|
| 执行中任务被取消 | 先完成当前步骤，再执行补偿 | 不进行重试 |
| 执行失败且未取消 | 先重试，重试失败后若需补偿则执行补偿 | 重试优先 |
| 补偿失败 | 可重试补偿，最多64次 | 补偿失败后状态保持FAILED |

### 5.4 重试与补偿退避策略

**任务重试退避策略**：
| 重试次数 | 延迟时间 | 计算公式 |
|---------|---------|---------|
| 第1次 | 5秒 | base_interval * 2^0 |
| 第2次 | 10秒 | base_interval * 2^1 |
| 第3次 | 20秒 | base_interval * 2^2 |
| ... | ... | ... |

其中任务重试base_interval默认5000ms（可配置）

**补偿重试退避策略**：
| 重试次数 | 延迟时间 | 计算公式 |
|---------|---------|---------|
| 第1次 | 1秒 | base_interval * 2^0 |
| 第2次 | 2秒 | base_interval * 2^1 |
| 第3次 | 4秒 | base_interval * 2^2 |
| ... | ... | ... |

其中补偿重试base_interval默认1000ms（可配置）

## 6. 数据规范需求

### 6.1 任务状态定义

| 状态 | 说明 | 可转换状态 |
|------|------|-----------|
| PENDING | 待执行 | RUNNING, CANCELED |
| RUNNING | 执行中 | SUCCESS, FAILED, CANCELED |
| SUCCESS | 执行成功 | - |
| FAILED | 执行失败 | PENDING(重试) |
| CANCELED | 已取消 | - |

### 6.2 补偿状态定义

| 状态 | 说明 |
|------|------|
| NONE | 无需补偿 |
| PENDING | 待补偿 |
| RUNNING | 补偿执行中 |
| SUCCESS | 补偿成功 |
| FAILED | 补偿失败 |

### 6.3 动作类型定义

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| SYNC | 同步执行 | 需要立即返回结果的场景 |
| ASYNC | 异步执行 | 默认模式，数据库拉取执行 |

### 6.4 回执类型定义

| 类型 | 说明 | 配置参数 |
|------|------|---------|
| HTTP | HTTP回执 | url, headers, bodyTemplate, retryPolicy, timeoutMs |
| MQ | MQ回执 | topic, keyTemplate, headers, bodyTemplate |
| NONE | 不回执 | - |

### 6.5 任务编号规则

**格式**：`TASK{yyyyMMddHHmmss}{毫秒内序号}{随机字符}`

**示例**：`TASK2026050212345612345A3F`

## 8. 错误码规范需求

### 8.1 错误码定义

| 错误码 | 错误信息 | 说明 |
|--------|----------|------|
| 10001 | API不存在 | apiCode对应的API配置不存在 |
| 10002 | API已禁用 | apiCode对应的API未启用 |
| 10003 | 参数校验失败 | 请求参数不符合过滤规则 |
| 10004 | 限流触发 | 触发限流规则 |
| 10005 | 系统繁忙 | 系统负载过高，请稍后重试 |
| 10006 | 任务不存在 | taskNo对应的任务不存在 |
| 10007 | 任务状态不允许此操作 | 当前状态不支持该操作 |
| 10008 | 插件不存在 | 指定的插件类不存在 |
| 10009 | 执行超时 | 任务执行超时 |
| 10010 | 执行失败 | 任务执行失败 |
| 10011 | 插件执行失败 | 插件处理过程中发生错误 |
| 10012 | 补偿超时 | 补偿执行超时 |
| 10013 | 补偿失败 | 补偿执行失败 |

### 8.2 错误响应格式

```json
{
    "success": false,
    "error": {
        "code": "10004",
        "message": "限流触发，请稍后重试",
        "detail": "当前QPS: 150, 限制QPS: 100"
    }
}
```

## 13. 告警接口埋点

系统预留告警埋点，在关键业务事件发生时触发，使用者可根据需要自行实现告警通知：

### 13.1 告警埋点事件类型

| 事件类型 | 说明 | 触发条件 |
|---------|------|---------|
| 任务异常告警 | 任务执行失败且重试次数用尽 | retry_count >= max_retry_count |
| 执行超时告警 | 任务执行超时 | status=RUNNING 且超过 runningTimeoutMinutes |
| 补偿失败告警 | 任务补偿执行失败 | compensate_status=FAILED |
| 限流告警 | 限流触发 | rate_limit_reject 触发 |
| 系统异常告警 | 系统内部异常 | 未捕获的异常 |
| 配置变更通知 | 配置变更通知 | 配置变更事件 |

### 13.2 告警埋点数据结构

```json
{
  "eventType": "TASK_FAILED",
  "eventTime": 1746153600000,
  "level": "WARNING",
  "message": "任务执行失败",
  "detail": {
    "taskNo": "TASK2026050212345612345A3F",
    "apiCode": "AMAZON_ORDER_QUERY",
    "status": "FAILED",
    "errorMessage": "插件执行异常"
  }
}
```

### 13.3 告警埋点扩展点

系统预留以下扩展点，使用者可根据需要自行实现：
- `AlarmSender` 接口：告警发送器
- `AlarmRuleEngine` 接口：告警规则引擎
- `AlarmStorage` 接口：告警存储
